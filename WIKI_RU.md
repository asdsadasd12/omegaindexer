# OmegaIndexer Tool Wiki

## Назначение

Этот Streamlit-инструмент помогает:

- собирать внутренние страницы сайта с главной или другой стартовой страницы;
- автоматически фильтровать страницы перед отправкой;
- отправлять URL в `OmegaIndexer` через API;
- при необходимости создавать отдельную кампанию под каждый домен.

Инструмент рассчитан на работу с одним или несколькими сайтами сразу.

## Основные режимы

В приложении есть 2 режима:

- `Collect from homepage links`
- `Manual URL submission`

### Collect from homepage links

Это основной режим.

Что делает:

1. Берет список сайтов.
2. Заходит на стартовую страницу каждого сайта с `Googlebot` user-agent.
3. Собирает внутренние ссылки.
4. Убирает дубли.
5. Проверяет страницы по правилам фильтрации.
6. Делит результаты на `Sitemap pages` и `Other pages`.
7. Отправляет выбранные URL в `OmegaIndexer`.

### Manual URL submission

Режим для ручной отправки подготовленного списка URL без автоматического сбора ссылок.

## Какие данные нужно вводить

### В боковой панели

- `OmegaIndexer API key`
- `Drip-feed days`
- `Homepage fetch timeout (sec)`
- `Max pages to keep`
- `Apply page limit per domain`
- `Prioritize Sitemap Page URLs`

### В основном режиме сбора

- `Sites (one domain per line)`
- `Campaign name`
- `Create separate campaign for each domain`
- `Page path to parse`

## Что означает каждая настройка

### OmegaIndexer API key

API-ключ из вашего аккаунта `OmegaIndexer`.

### Drip-feed days

На сколько дней растягивать отправку URL в `OmegaIndexer`.

- `1` = отправлять сегодня, без растягивания.
- `7` = растянуть отправку примерно на 7 дней.

### Homepage fetch timeout (sec)

Сколько секунд ждать ответ от сайта при:

- открытии стартовой страницы;
- проверке найденных URL.

Если сайт не отвечает дольше этого времени, страница считается ошибочной.

### Max pages to keep

Максимальное количество страниц, которое останется после фильтрации.

По умолчанию: `50`.

### Apply page limit per domain

Если галочка включена, лимит страниц считается отдельно для каждого домена.

Пример:

- 2 сайта
- лимит `50`
- галочка включена

Результат:

- до `50` URL для первого домена;
- до `50` URL для второго домена.

Если галочка выключена, лимит будет общим на весь запуск.

### Prioritize Sitemap Page URLs

Если галочка включена:

- URL со `sitemap-page` получают приоритет;
- сначала в очередь попадают все `sitemap-page` по всем доменам;
- только потом остальные страницы.

Дополнительно инструмент автоматически пытается добавить URL вида:

`/top/sitemap-page-1/`

для каждого домена.

### Create separate campaign for each domain

Если галочка включена, инструмент отправляет отдельную кампанию для каждого домена.

Пример:

- `site1.com`
- `site2.com`

Будут созданы 2 отдельные кампании, а не одна общая.

Имя кампании формируется так:

`Campaign name - domain.com`

### Page path to parse

Стартовая страница, с которой начинать сбор ссылок.

По умолчанию:

`/`

Это главная страница сайта.

Можно указать и другой путь, например:

- `/blog/`
- `/top/`
- `/category/news/`

## Как работает фильтрация страниц

После сбора ссылок инструмент оставляет только подходящие страницы.

Страница проходит фильтр, если:

- отдает `200 OK`;
- не редиректит на другой финальный URL;
- canonical либо указывает на саму себя, либо отсутствует;
- страница не закрыта от индексации через `meta robots`, `googlebot` или `X-Robots-Tag`.

Страница исключается, если:

- код ответа не `200`;
- есть редирект на другой URL;
- canonical указывает на другую страницу;
- есть `noindex` или аналогичный запрет индексации;
- страница не ответила в установленный таймаут.

## Как работает многопоточность

Проверка найденных URL идет в `10` потоков.

Это ускоряет проверку:

- кода ответа;
- canonical;
- доступности для индексации.

Многопоточность используется только на этапе проверки найденных страниц.

## Как работает приоритет sitemap при нескольких сайтах

При включенном приоритете логика такая:

1. Собираются все валидные `sitemap-page` URL со всех доменов.
2. Они ставятся в начало очереди.
3. После них добавляются остальные валидные страницы.
4. Затем применяется лимит.

Если лимит на домен включен, он применяется отдельно для каждого домена, но итоговая очередь все равно сначала заполняется `sitemap-page`, потом остальными страницами.

## Что показывается в результатах

После парсинга интерфейс показывает:

- сколько ссылок найдено по каждому сайту;
- сколько дублей удалено;
- сколько URL было до лимита;
- сколько прошло фильтр;
- какой лимит применен;
- сколько URL выбрано на отправку.

Также есть 3 блока:

- `Sitemap pages`
- `Other pages`
- `Excluded pages`

### Sitemap pages

Список валидных sitemap-страниц, которые прошли проверку.

### Other pages

Список остальных валидных страниц.

### Excluded pages

Список URL, которые не прошли фильтр, с причиной исключения.

Пример:

- `[404] https://site.com/page -> HTTP status is not 200`
- `[200] https://site.com/page -> Canonical points to another URL`

## Как работает отправка в OmegaIndexer

Инструмент отправляет кампании в API `OmegaIndexer`.

Используются поля:

- `apikey`
- `campaignname`
- `urls`
- `dripfeed`

Поле `urls` отправляется одной строкой через разделитель `|`.

Пример:

`https://site.com/a|https://site.com/b|https://site.com/c`

## Типовой сценарий работы

1. Введите `API key`.
2. Убедитесь, что `Drip-feed days = 1`, если отправка нужна сегодня.
3. Укажите лимит страниц.
4. Вставьте один или несколько доменов.
5. Оставьте путь `/`, если нужно парсить главную.
6. Нажмите `Fetch homepage links`.
7. Проверьте блоки `Sitemap pages`, `Other pages`, `Excluded pages`.
8. При необходимости скорректируйте выбранные URL.
9. Нажмите `Send collected URLs to OmegaIndexer`.

## Важные особенности

- Лимит применяется после фильтрации.
- Duplicates удаляются до отправки.
- Приоритет `sitemap-page` не означает, что такие URL автоматически индексируемы: они тоже проходят все проверки.
- Если один и тот же URL найден несколько раз, в итоговый список он попадает один раз.
- Если включено создание кампаний по доменам, каждый домен отправляется отдельно.

## Возможные причины, почему URL не попал в отправку

- страница вернула не `200`;
- canonical указывает на другой URL;
- есть `noindex`;
- сработал таймаут;
- URL не вошел в лимит;
- URL был дублем;
- URL не был выбран в списке перед отправкой.
